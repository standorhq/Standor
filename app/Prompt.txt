Your task: Design and implement a production-grade live coding interview platform called **Standor** using a **MERN + TypeScript** stack and real-time collaboration. The output must be a professional, industry-standard web application suitable for large tech companies.

You are allowed to generate:
- Full repository structure (frontend + backend).
- Source code files.
- Configuration files (Docker, Kubernetes manifests, ESLint/Prettier, CI).
- Schema and API definitions.
- Detailed comments and minimal documentation where needed.

You are NOT allowed to:
- Change the core product scope or feature set.
- Switch major technologies without very strong reasons; prefer to stay within the stack below.

=====================================================
1. GLOBAL REQUIREMENTS & QUALITY BAR
=====================================================

Role & standards:
- Act as a principal engineer at a FAANG company.

Backend:
- Node.js + Express + TypeScript.

Code execution:
- Use **Piston API v2** for code execution (`/api/v2/execute`) for all supported languages.

AI integration:
- Use HTTP APIs for **DeepSeek / OpenRouter / Ollama**.
- Design AI integration so providers can be swapped by configuration.
- AI usage:
  - Post-interview static analysis.

Infra & DevOps:

  - Frontend deployment.

3. CORE DOMAIN MODELS (MONGOOSE)
=====================================================

Design TypeScript interfaces + Mongoose schemas/models for at least:

- User
  - _id, email, passwordHash, name.
  - roles: [ "recruiter", "interviewer", "candidate", "admin" ].
  - orgId (Organization reference).
  - login metadata (lastLoginAt, failedLoginCount, lockoutUntil, createdAt, updatedAt).

- Organization
  - _id, name, slug, settings (RBAC rules, branding options, quotas).
  - createdAt, updatedAt.

- InterviewRoom
  - _id, orgId, title, company, difficulty, durationMinutes.
  - primaryLanguage, languagesAllowed.
  - problemIds used in this room.
  - status: "waiting" | "live" | "paused" | "completed" | "archived".
  - scheduledStartAt, startedAt, endedAt.
  - participants: [{ userId, role, joinedAt, leftAt }].
  - settings: flags like allowHints, recordEvents, recordVideoMetadataRef (not actual video file).
  - createdBy (recruiter or interviewer).

- Problem
  - _id, orgId, title, description (Markdown/HTML-safe).
  - difficulty, tags.
  - visibleTests: [{ input, expectedOutput, explanation? }].
  - hiddenTests: [{ input, expectedOutput }].
  - defaultLanguage, starterCode per language.
  - attachmentUrl(s) for PDFs / diagrams.
  - createdBy, createdAt, updatedAt.

- CodeSnapshot
  - _id, roomId, orgId, userId (who triggered save).
  - files: [{ name, language, content }].
  - createdAt.
  - isFinal (for final snapshot when interview ends).

- Message
  - _id, roomId, orgId, senderId.
  - type: "text" | "code" | "system".
  - body (text), codeSnippet?, language?.
  - createdAt.
  - systemType? (“join”, “leave”, “ai-analysis-complete”, etc).

- Analysis (AI Reports)
  - _id, roomId, orgId, candidateId.
  - problemId(s).
  - provider: "deepseek" | "openrouter" | "ollama".
  - rawResponse (JSON).
  - structured:
    - timeComplexityBefore, timeComplexityAfter.
    - spaceComplexity.
    - bugList: [{ description, severity, suggestedFix }].
    - styleScore, readabilityScore.
    - improvementHints: string[].
  - humanNotes: [{ recruiterId, note, createdAt }].
  - createdAt, updatedAt.

- EventLog
  - _id, roomId, orgId.
  - events: append-only array or compressed segments:
    - type: "editor-delta" | "cursor" | "file-change" | "chat" | "room-state" | "system".
    - payload (strongly typed TS union).
    - timestamp, actorId.

- AuditLog
  - _id, orgId.
  - action: "role-change" | "ai-rerun" | "recording-download" | "org-settings-change" | etc.
  - actorId, targetId, targetType (User/Org/Room/etc).
  - metadata, createdAt.
  - immutable (no updates/deletes).

You should:
- Provide Mongoose schema definitions with TypeScript typings.
- Split model definitions into `backend/src/models/*.ts`.
- Expose reusable TypeScript domain types in `backend/src/types/domain/*.ts` for sharing between layers.

4. AUTH, USERS, ORGANIZATIONS
=====================================================

Auth flows:
- Email+password signup/login with hashed passwords (bcrypt or argon2).
- Google OAuth login using Passport or a custom OIDC client.
- JWT-based or signed-cookie-based sessions (choose one and justify briefly in comments).
- Middleware for:
  - `requireAuth`.

Security:
- Rate limiting on auth endpoints (Redis-backed counters; per IP + per user).
- Account lockouts after configurable failed attempts.
- Security-related audit logs (password change, new login, failed lockouts).
- CSRF and CORS properly configured for SPA + API setup.

Org multi-tenancy:
- All major resources scoped by `orgId`.
- Ensure queries always filter by `orgId` derived from authenticated user and/or route parameters.

API endpoints (Express):
- `POST /api/auth/signup`
- `POST /api/auth/login`
- `POST /api/auth/logout`
- `POST /api/auth/refresh` (if JWT refresh tokens are used)
- `GET  /api/users/me`
- `GET  /api/orgs` (for admins)
- `POST /api/orgs` (create org, admin only)
- `PATCH /api/orgs/:orgId` (admin only)

5. INTERVIEW ROOMS & LIFECYCLE
=====================================================

Interview room CRUD:
- Recruiters/interviewers:
  - `POST   /api/rooms`
  - `GET    /api/rooms`
  - `GET    /api/rooms/:roomId`
  - `PATCH  /api/rooms/:roomId`
  - `DELETE /api/rooms/:roomId` (soft-delete or archive)
- Include fields: title, company, difficulty, duration, languages, problems, scheduled time, settings.

Join links & lobby:
- Candidate/interviewer join via unique URL: `/room/:roomId`.
- Backend validates:
  - Room exists and belongs to their organization.
  - User has a valid role and is invited (if invite-only).
- Lobby screen behavior:
  - Mic/cam permissions + preview (WebRTC getUserMedia).
  - Latency ping using Socket.IO.
  - “Ready” toggle per side, synced to backend.

Room states:
- State machine stored on InterviewRoom:
  - "waiting" -> "live" -> ("paused" <-> "live") -> "completed" -> "archived".
- Enforce transitions in service layer; invalid transitions should be rejected.
- Socket.IO broadcasts state changes to all participants.

6. REAL-TIME CODE COLLABORATION (MONACO + SOCKET.IO)
=====================================================

Editor:
- React wrapper around Monaco Editor in TypeScript mode by default.
- Support multiple languages (JS/TS, Python, Java, C++, etc.) based on room/problem config.

Document syncing:
- Use Yjs or a simple OT/CRDT layer on the frontend for local collaborative state.
- Over Socket.IO:
  - Room-level namespace, e.g., `/room/:roomId`.
  - Events:
    - `code:init` (initial snapshot).
    - `code:delta` (editor changes).
    - `code:fileChange` (switching active file).
    - `presence:cursor` (cursor positions/selections).
    - `presence:status` (joined/left/ready).

Server behavior:
- Primarily broadcast and store periodic snapshots (no heavy OT logic on server).
- Periodic or explicit snapshot save route:
  - `POST /api/rooms/:roomId/snapshots` -> CodeSnapshot.

Multiple files:
- Room-level "file tree":
  - Maintained in memory and persisted via snapshots.
  - Supports add/delete/rename file (within room session).
- Sync current file selection and file list across participants.

Autosave:
- Frontend triggers:
  - Every 30 seconds.
  - On explicit “Save now”.
- Backend stores snapshots in CodeSnapshot collection.

=====================================================
7. CODE EXECUTION (PISTON API)
=====================================================

Run code:
- Frontend:
  - “Run” button sends:
    - roomId, problemId, selected file(s), language, stdin (sample input).
- Backend:
  - Validates user access and room state.
  - Maps requested language to Piston runtime/language.
  - Calls Piston `/api/v2/execute` with given code/files and stdin.
  - Stores raw execution result for analytics (not full stdout if too large; consider truncated).
  - Returns stdout, stderr, runtime, and exit code.

Test cases:
- `Problem` documents contain visible and hidden tests.
- Backend run endpoint:
  - Executes code against:
    - Visible tests: returned to both candidate and interviewer.
    - Hidden tests: detailed results only visible to interviewer/backend analytics.
- Design:
  - `POST /api/rooms/:roomId/problems/:problemId/run` for running tests.

Execution panel:
- React console panel:
  - Shows stdout, stderr, runtime, memory (if available), and per-test pass/fail.
  - Clear separation of visible vs summary stats (hidden).

=====================================================
8. AI CODE ANALYSIS (DEEPSEEK / OPENROUTER / OLLAMA)
=====================================================

Post-interview analysis:
- When a room transitions to "completed":
  - Enqueue a BullMQ job on an `ai-analysis` queue.
  - Worker:
    - Loads final CodeSnapshot + Problem details + run history.
    - Sends structured prompt to configured AI provider.
    - Parses response into `Analysis` document.
    - Emits a system chat event “AI analysis ready”.

AI providers:
- Abstract provider interface:
  - `analyzeSolution({ code, language, problem, runStats }): Promise<AnalysisResult>`.
- Concrete implementations for:
  - DeepSeek via HTTP.
  - OpenRouter (router to different model).
  - Ollama (local or remote instance).
- Provider selection by config environment variable.

AI panel (frontend):
- In Room “Review” view:
  - Show complexity (before/after if applicable), bug count, style score, improvement hints.
  - Allow recruiters to:
    - Add human notes.
    - Override or supplement AI scores.

Optional live hints:
- UI toggle for interviewer: “AI hints enabled”.
- Backend periodic job:
  - At intervals (e.g., every N seconds/minutes) when enabled:
    - Sends current snapshot + problem summary to AI provider.
    - Returns brief non-spoiler hint.
  - Stream hints to “AI hints” sidebar via Socket.IO.
