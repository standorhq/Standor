Software Defined Access (SDA) is Cisco’s campus architecture that separates policy from traditional device-by-device configuration.It uses a controller-based approach where Cisco DNA Center manages and automates the entire fabric network.
The SDA architecture consists of three main components: DNA Center, the Fabric infrastructure, and Cisco ISE.
The underlay network is a pure Layer 3 routed IP network that provides basic reachability between all fabric devices.
Routing protocols like IS-IS or OSPF are used in the underlay to ensure fast convergence and scalability.
The overlay network runs on top of the underlay using VXLAN encapsulation for traffic separation.
VXLAN adds an outer IP header and a VNI, which allows creation of up to 16 million logical networks.
The control plane in SDA uses LISP to map endpoint identity (EID) to device location (RLOC).
LISP allows endpoint mobility without changing IP addresses because location and identity are separated.
Fabric Edge Nodes connect users and perform VXLAN encapsulation and decapsulation of traffic.
Control Plane Nodes maintain the endpoint database and respond to mapping queries from edge devices.
Border Nodes connect the SDA fabric to external networks like the data center or internet.
Cisco ISE performs 802.1X authentication and acts as the RADIUS server for identity verification.
After authentication, ISE assigns a Security Group Tag (SGT) to the user or device.
SGT enables micro-segmentation by enforcing policy based on identity instead of IP addresses.
Macro-segmentation is achieved using Virtual Networks, which isolate different departments or services.
Policy enforcement is done using SGACLs, which define communication rules between security groups.
DNA Center automates device onboarding, fabric provisioning, and policy deployment across the campus.
SDA removes VLAN sprawl, reduces manual ACL configuration, and eliminates dependence on spanning tree.
Overall, SDA provides scalable, secure, automated, and identity-based networking for modern enterprise campuses.
 
Cisco SD-Access (SDA)-  Research lines Focus Areas: DNAC, Fabric, ISE
Cisco SD-Access is Cisco's intent-based networking solution built on the Digital Network Architecture.
The three core pillars are the Management Plane (Catalyst Center), the Fabric (overlay + underlay), and the Policy Plane (ISE). The power of SDA comes from tight integration of all three and none of them works effectively in isolation.
Catalyst Center (formerly DNAC) is the single pane of glass for the entire SDA ecosystem. It handles Day-0 through Day-N operations including device provisioning via Plug-and-Play, software image management, policy authoring, and AI-powered assurance.
One underappreciated DNAC feature is seed-switch-based onboarding. Only one device needs to be active on the dashboard and all other switches pull their underlay settings automatically from that seed.
The SDA Fabric has two layers, Underlay and Overlay. The underlay is a routed Layer 3 network using IS-IS or OSPF, with BFD-enabled ECMP links and jumbo MTU (typically 9100) to handle VXLAN overhead.
The overlay uses VXLAN for data plane encapsulation and LISP as the control plane.
 
The Control Plane Node maintains the EID-to-RLOC mapping database, tracking which endpoint sits behind which edge node.
Fabric roles are well defined. Edge Nodes are anycast gateways at the access layer, Border Nodes connect the fabric to external networks, and Control Plane Nodes run the LISP map-server.
SDA delivers two segmentation layers. Macrosegmentation uses Virtual Networks (VNs) backed by dedicated VRFs and VXLAN VNIs to hard-isolate traffic groups like IoT from corporate users. Microsegmentation uses Security Group Tags (SGTs) embedded in the VXLAN header to enforce identity-based policy within a VN without touching IP addressing.
SGTs follow the user or device regardless of where they connect. Policies are defined as a Source SGT to Destination SGT matrix with Permit or Deny decisions.
ISE is the policy brain of SDA. It authenticates users and devices via 802.1X, assigns SGTs based on role, device posture, and location, and communicates those bindings to the fabric.
ISE integrates with Catalyst Center via pxGrid for real-time SGT binding events and ERS REST APIs for bulk policy synchronization.
One operational detail that trips teams up is that Catalyst Center supports only a single ISE Policy Admin Node (PAN) integration at a time.
SDA wireless is native and consistent. Fabric-mode APs register directly with the LISP control plane via the Fabric WLC. Wireless clients get the same SGT treatment as wired clients, giving uniform policy enforcement across connection types, something traditional NAC solutions always struggled with.
From a Zero Trust standpoint, SDA is Cisco's campus implementation of Zero Trust.
DNAC Assurance correlates streaming telemetry, ISE authentication events, and NetFlow data using AI/ML to pinpoint specific issues like a client experiencing roaming latency due to an SGT mismatch.
Palo Alto NGFWs complement SDA at the border. SDA handles East-West microsegmentation within the campus via SGTs while Palo Alto provides Layer 7 inspection and threat prevention for North-South traffic at the fabric border. Many enterprises route inter-VN traffic through a Palo Alto firewall for full application-layer visibility.
Palo Alto Prisma SASE extends this further to remote users. Prisma Access via GlobalProtect gives remote workers the same identity-driven access policies as on-campus users. Cisco has a validated integration guide for Catalyst SD-WAN and Palo Alto Prisma SSE using IKEv2 tunnels for cloud-based NGFW enforcement at branches.
SDA adoption numbers are telling. As of CiscoLive 2024, over 24,000 sites with 1.8 million+ devices are under management across 4,000+ installations.
The hardest part of SDA is the migration. Moving from usual VLAN-based networks means rethinking IP pools, DHCP scopes, and policy design from scratch. Teams that skip proper design and go straight to the wizard are the ones who struggle.

In traditional networking, devices have three main planes: Data Plane, Control Plane, and Management Plane.
The Data Plane is responsible for forwarding packets from source to destination at high speed.
The Data Plane works mostly in hardware and enforces policies while forwarding traffic.
The Control Plane decides the path traffic should take inside the network.
The Control Plane maintains routing tables and endpoint mapping information.
The Management Plane is used by network administrators to configure, monitor, and manage devices.
In SD-Access, these planes are logically separated and centrally managed.
SD-Access is part of Cisco’s Intent-Based Networking architecture from Cisco Systems.
The main controller of SD-Access is Cisco DNA Center.
Cisco DNA Center provides centralized automation, configuration, and monitoring.
The SD-Access network works using a fabric architecture.
The fabric consists of an Underlay (physical network) and an Overlay (virtual network).
The Underlay provides basic IP connectivity between network devices.
The Overlay uses VXLAN to create virtual networks and provide segmentation.
The Control Plane in SD-Access uses LISP to map device identity to its location.
LISP separates EID (Endpoint ID) from RLOC (Routing Locator).
The Data Plane uses VXLAN encapsulation to transport traffic across the fabric.
Security and policy enforcement are handled by Cisco TrustSec technology.
User authentication and policy decisions are done by Cisco Identity Services Engine (ISE).
Overall, SD-Access simplifies enterprise networking by providing automation, security, segmentation, and centralized control.
